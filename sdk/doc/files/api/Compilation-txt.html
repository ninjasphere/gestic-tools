Windows

The GestIC SDK installation provides an API implementation and a few Windows applications (App) as source code together with precompiled binaries.

It is possible to start a new application directly based on those binaries.

Nevertheless here are instructions for building the API and the Apps.

Building the API

As the installation directory is normally write protected it is recommended to copy api/include and api/src to a separate development folder before building.

The src directory of this copy contains a gestic-api.sln solution file that could be used with Microsoft Visual C++ 2010.

This solution provides multiple configurations (Dynamic-Debug, Dynamic-Release, Static-Debug and Static-Release) that differ for the built API (Static API vs Dynamic API from the API Usage page contains details on this) and for whether they are for debugging or release.

Depending on the configuration this project builds standalone gestic.lib or a combo of the gestic.dll and an associated gestic.lib import library.

Building the Apps

In order to build an App the directory of the App has to be copied to an development directory where it could be built.  Each App has its own solution file that could be used with Microsoft Visual C++ 2010.

The Apps build and link against the prebuild binaries in the installation directory.

Linux

This package also contains the source code for building the SDK on Linux.  This requires the content of the following folders:

  • apps/Linux
  • api/include
  • api/src

The directory apps/Linux contains a Makefile that build the API and the Linux apps.  This Makefile expects gcc as the compiler and ar for greating lib-files.  The console-application also requires a curses-compatible library to be installed.

On Ubuntu 14.04 the required packages could be installed with:

sudo apt-get install build-essential libncurses5-dev

Once all requirements are installed the project could be build by going the the apps/Linux directory and running make there.

cd apps/Linux
make

The Makefile currently doesn’t contain instructions for installation.  However the final binaries are located in apps/Linux/build/bin after building finished successfully.

The GestIC expects the GestIC Bridge to be the device at /dev/ttyACM0.  This behaviour could be changed in the following line in api/src/io/cdcserial_linux.c:

#define DEVICE "/dev/ttyACM0"

By default only root and users in the dialout group have access to GestIC Bridges device file.  To privilege normal users to interact with the device a udev rule is required.  The following rule changes the group for this device to plugdev and also gives all users read and write access.  It also creates a symbolic link for the device at /dev/gestic for convenience.

SUBSYSTEMS=="usb", ATTRS{idVendor}=="04d8", ATTRS{idProduct}=="000a", SYMLINK+="gestic", GROUP="plugdev", MODE="0666"

This rule could be installed by typing the following command into a terminal window

sudo sh -c 'echo "SUBSYSTEMS==\"usb\", ATTRS{idVendor}==\"04d8\", ATTRS{idProduct}==\"000a\", SYMLINK+=\"gestic\", GROUP=\"plugdev\", MODE=\"0666\"" >> /etc/udev/rules.d/99-gestic.rules'

The new udev rule could either be loaded by a system restart or with the following commands:

sudo udevadm control --reload-rules
sudo udevadm trigger --attr-match=idVendor="04d8" --attr-match=idProduct="000a"

PIC32

Each app in apps/PIC32 contains an MPLAB X Project that allows to compile the app.  There is no common API library for the PIC32.  Instead each app includes all necessary files.
